# AdamRogue 阶段C实现规范

## 文档定位

本文档用于指导开发者 agent 实现阶段 C。

阶段 C 的目标不是做完整装备系统，而是把当前玩法闭环扩展为：

`单位奖励 -> 战斗 -> 装备奖励`

并确保该奖励段能够被稳定测试、读档恢复与日志排查。

## 阶段目标

本阶段要完成 5 件事：

1. 在玩家战斗胜利后，进入“装备奖励待领取”状态。
2. 玩家可以在候选装备中做一次选择。
3. 选择结果进入派系库存，而不是自动穿到角色身上。
4. 奖励质量与 run 进度存在基础关联，而不是完全纯随机。
5. 玩家完成装备选择后，自动进入下一轮单位奖励事件。

## 本阶段边界

本阶段只做“装备奖励 v1”，不做以下内容：

- 不做虚拟地图或目的地节点。
- 不做装备自动穿戴推荐。
- 不做复杂锻造、拆解、商店、出售功能。
- 不做全派系通用装备池。
- 不做旗帜、追随者类奖励。

## 奖励触发规则

第一版按下面规则处理：

- 仅在玩家获得战斗胜利后，将流程推进到“装备奖励待领取”状态。
- 战斗结束后不立即自动弹出装备奖励事件。
- 玩家需要手动点击主入口，才进入本轮装备奖励选择。
- 战斗失败不触发装备奖励，直接进入既有失败处理逻辑。
- 单位奖励事件仍作为战斗前 build 选择。
- 装备奖励事件位于战斗结算之后、下一轮单位奖励之前。

结论：

- 阶段 C 是对当前闭环的补全，不改写阶段 B 的胜败判定主体。

## 装备奖励类别范围

第一版只允许以下装备类别进入候选池：

- `weapon`
- `armour`
- `talisman`
- `enchanted_item`
- `arcane_item`


这意味着：

- 可以出现重复物品
- 不强制排除任务专属物品
- 不强制排除世界唯一或强绑定唯一物品

实现要求：

- 第一版以“先跑通装备奖励闭环”为主。
- 若后续测试证明某些物品类型会导致实际发放、显示或装备行为异常，再单独回收限制。

## 稀有度规则

第一版允许使用当前游戏可用的全部稀有度，不预先做人为裁剪。

建议进度映射：

- 前期战斗以 `common` 为主。
- 中期开始开放 `uncommon`。
- 更后期再逐步提高 `rare` 出现概率。
- 若数据中存在更高或特殊稀有度，可在更后阶段逐步纳入。

实现要求：

- 稀有度不能完全独立于进度。
- 稀有度推进可以基于 `completed_battle_count`，也可以基于已有难度阶段字段。
- 若现有代码已经有 battle tier / difficulty level 字段，优先复用，不新增平行概念。

## 奖励池组织方式

第一版不建议直接对全量 ancillary 数据库做无筛选随机。

建议方案：

- 维护一份明确的装备候选池配置。
- 每条候选记录至少应包含：
  - `item_key`
  - `item_category`
  - `item_rarity`
  - `weight`
  - `min_battle_tier`
  - `max_battle_tier`
  - `enabled`


设计理由：

- 第一版最重要的是可控、可测、可排错。
- 显式池配置比“盲抽全库”更容易定位异常物品与后续平衡问题。

## 候选生成规则

建议第一版固定目标为 3 个候选。

如果池过小，可以降级为：

- 有几个候选就给几个候选
- 不强行补满

生成要求：

1. 不要求同一事件内对 `item_key` 去重。
2. 不要求因派系已拥有该物品而重掷。
3. 若最终候选数少于目标值，直接按实际数量展示，并写日志。

建议保留的生成上下文字段：

- `selected_rarity_band`
- `candidate_generation_seed`
- `candidate_generation_attempts`
- `fallback_strategy_used`

## 奖励发放方式

第一版目标是“进入派系库存”。

要求：

- 选择结果应发放到 faction inventory。
- 不直接绑到某个角色。

可接受实现方式：

- 直接调用向派系发放 ancillary 的 API
- 或通过 payload / incident 路径发放

## 状态机与领取语义

阶段 C 不提供“稍后再选”。

建议新增或明确的状态：

- `EQUIPMENT_REWARD_PENDING`

状态含义：

- `EQUIPMENT_REWARD_PENDING` 表示玩家战斗胜利后，尚未手动领取本轮装备奖励。
- 玩家点击主入口时，若当前状态是 `EQUIPMENT_REWARD_PENDING`，则应进入装备奖励选择，而不是直接开始下一轮单位奖励。
- 玩家完成装备选择后，应自动推进到下一轮单位奖励事件。

存档/读档要求：

- 若玩家已经打完战斗但尚未点开装备奖励，读档后仍应保持 `EQUIPMENT_REWARD_PENDING`。
- 不能因为重载脚本而跳过该奖励


## 本地化与文本要求

本阶段会新增一组装备奖励事件文本。

最低要求：

- 事件标题不是空白
- 事件正文不是空白
- 每个候选选项有可读文本

文案第一版不追求大量风味文本，但必须保证可测试性：

- 玩家能看懂自己拿到的是哪一类装备
- 玩家能看懂这是本轮装备奖励，而不是下一轮单位奖励

## 日志要求

阶段 C 必须继续输出独立日志，便于测试时排查。

至少记录：

- 战斗胜利后是否进入装备奖励流程
- 战斗胜利后是否正确切到 `EQUIPMENT_REWARD_PENDING`
- 本次目标稀有度区间
- 候选物品列表
- 候选生成时采用的池与条件
- 因候选不足触发的 fallback
- 玩家最终选择
- 奖励成功发放结果
- 选择完成后是否正确推进到下一轮单位奖励
- 存档恢复行为

结论：

- 日志应足够让人判断“为什么出现这 3 个候选”以及“为什么最终拿到这个物品”。

## 开发者实现约束

开发者 agent 在阶段 C 中应遵守：

1. 优先复用原版已有 ancillary / reward API，不手写奇怪的替代发放逻辑。
2. 先把候选池做小而稳，不为了“全量内容”牺牲可验证性。
3. 不要把阶段 D 的地图推进提前混入本阶段提交。
4. 若发现某些物品 key 无法正常发放，应记录具体 key 与异常表现，再决定是否局部排除。
5. 若现有角色或派系限制导致某类装备无法进入库存，优先记录问题并做最小必要修正，不要临时引入大规模额外系统。
6. 本阶段不提供“稍后再选”按钮，也不实现其相关状态分支。

## 完成标准

阶段 C 完成后，应满足下面这些可观察结果：

1. 玩家首次打赢战斗后，不会立刻自动弹出奖励，而是进入待领取装备奖励状态。
2. 事件中能看到非空标题、正文与候选按钮文本。
3. 玩家选择任一装备后，该装备进入派系库存。
4. 该装备不会被自动装备到角色身上。
5. 玩家手动点击主入口后，才能进入本轮装备奖励选择。
6. 玩家完成装备选择后，会自动进入下一轮单位奖励事件。
7. 存档并读档后，仍然是同一组选项或同一待领取状态。
8. 候选允许出现重复物品，也允许出现任务专属或唯一物品。
9. 当候选池不足时，事件按实际数量展示选项。
10. 日志中能看出候选生成、发放、状态推进与恢复流程。

## 对开发者 agent 的交付说明

开发者 agent 在正式编码前，应把实现理解锁定为：

- 阶段 C 不是“做装备系统大全”
- 阶段 C 是“做战后装备奖励闭环 v1”

如果实现过程中出现分歧，优先遵守以下排序：

1. 可测试
2. 可恢复
3. 可排错
4. 再考虑扩展性

## 后续阶段衔接

阶段 C 完成后，阶段 D 再进入：

- 虚拟地图
- 目的地节点
- 路线推进
- 更完整的 run 节奏控制

也就是说，阶段 C 的职责是先把“战后拿装备”这一步稳定下来，为后续地图层提供一个可靠中间段。
