# AdamRogue 阶段D实现规范

## 文档定位

本文档用于指导开发者 agent 实现阶段 D。

阶段 D 的目标不是做真实世界地图移动系统，而是先做一个足够稳定的“派系节点式虚拟地图 v1”。

阶段 D 完成后，玩法闭环应扩展为：

`单位奖励 -> 战斗 -> 装备奖励 -> 目的地选择 -> 下一轮单位奖励`

## 阶段目标

本阶段要完成 6 件事：

1. 建立“地图节点 = 主要派系”的虚拟地图模型。
2. 在流程中加入目的地选择事件。
3. 记录并维护“当前所处节点派系”。
4. 让敌军生成逻辑能够读取当前节点派系，生成对应派系敌军。
5. 让装备奖励逻辑能够读取当前节点派系，按“通用 + 玩家派系 + 当前节点派系”生成候选。
6. 保持该系统支持手动推进、稍后再选、存档恢复与日志排查。

## 本阶段边界

本阶段只做“虚拟地图节点与派系上下文接管 v1”，不做以下内容：

- 不做真实地理位置移动。
- 不做节点相邻关系图。
- 不做距离、路径、航线、陆路、海路规则。
- 不做地图可视化 UI。
- 不做复杂的节点事件权重系统。
- 不强求本阶段完成“单位奖励第三选项 = 当前节点派系单位”。

最后一条说明：

- 用户已明确，如果单位奖励部分改动过大，可以在阶段 D 第一版中延后，只先完成敌军与装备奖励的派系接管。

## 节点模型设计

节点不代表坐标点，而代表“主要派系上下文”。

建议每个节点至少包含：

- `node_key`
- `faction_key`
- `culture_key`
- `display_name`
- `enabled`

设计结论：

- 节点的核心身份是“所属派系”。
- 不需要设计节点间连接关系。
- 同一个派系通常只需要一个节点。

## 节点候选范围

第一版应整理游戏内主要派系，作为节点候选池。

实现要求：

- 以“主要文化/主要派系节点池”优先。
- 具体收录范围应在配置中显式维护，而不是散落在脚本逻辑里。

## 目的地选择事件规则

每次进入目的地转移事件时，固定提供 4 个选项：

1. 从全部可用节点中随机抽出的候选节点 A
2. 从全部可用节点中随机抽出的候选节点 B
3. `留在当前派系`
4. `稍后再选择`

生成要求：

- 候选 A 与候选 B 不能相同。
- 候选 A/B 不应与当前节点相同；否则“留在当前派系”将失去意义。

第一版建议：

- 节点池如果足够，始终给出 2 个随机不同派系节点。
- `留在当前派系` 不改变当前节点。
- `稍后再选择` 保留当前状态，不推进到下一轮单位奖励。

## 目的地事件在流程中的位置

阶段 D 需要改写阶段 C 结束后的衔接。

旧流程：

`战斗 -> 装备奖励 -> 下一轮单位奖励`

新流程：

`战斗 -> 装备奖励 -> 目的地选择 -> 下一轮单位奖励`

具体要求：

- 战斗胜利后，仍然先进入装备奖励待领取状态。
- 玩家手动点击入口领取装备奖励。
- 玩家完成装备奖励选择后，立即进入目的地选择事件。
- 玩家完成目的地选择后，才进入下一轮单位奖励事件。

结论：

- 阶段 D 会替换阶段 C 中“装备奖励后直接进单位奖励”的行为。

## 状态机要求

建议新增或明确以下状态：

- `DESTINATION_PENDING`
- `DESTINATION_SELECTION_OPEN`

状态含义建议：

- `DESTINATION_PENDING`
  - 玩家已完成装备奖励，但尚未确定下一站。
  - 点击主入口时，应优先打开目的地选择事件。
- `DESTINATION_SELECTION_OPEN`
  - 当前目的地候选已生成，事件正在等待玩家选择。

与此相关的恢复语义：

- 若玩家选了“稍后再选择”，应保留本次目的地候选。
- 重新点击入口后，应看到同一组目的地选项，而不是重抽。
- 存档/读档后，也应保持同一组目的地候选。

## 持久化字段建议

阶段 D 至少应持久化以下信息：

- `current_node_key`
- `current_node_faction_key`
- `destination_candidate_node_keys`
- `destination_candidate_faction_keys`
- `destination_leave_current_enabled`
- `destination_selection_generated`

建议补充：

- `destination_generation_seed`
- `destination_generation_attempts`
- `destination_pending_from_state`

原则：

- 持久化的是“本次目的地候选结果”，不是只保存生成条件。
- 只要玩家已经进入本轮目的地选择，就不应在恢复时重新抽两个节点。

## 当前节点对后续系统的影响

本阶段最重要的不是“节点名字”，而是“节点派系上下文被真正接管”。

### 敌军生成

要求：

- 战斗敌军的候选池读取 `current_node_faction_key`。
- 阶段 B 的价值分生成器继续沿用。
- 当前变化仅是“单位池来源派系”切换到当前节点派系。

第一版允许：

- 仍保留旧的震旦池作为 fallback。
- 若某节点派系尚未建好单位池，则记录日志并回退到默认池。

### 装备奖励生成

要求：

- 装备奖励候选应由三类内容组成：
  - 通用装备池
  - 玩家派系装备池
  - 当前节点派系装备池

第一版允许：

- 若某节点派系尚未建好装备池，则只使用“通用 + 玩家派系”。
- 若玩家派系池暂时与通用池未严格拆分，也可先通过配置字段模拟来源归属。

### 单位奖励生成

用户当前期望是：

- 前两个选项固定为玩家派系单位。
- 第三个选项为当前节点派系单位。

同时需要提前调研后再实现的是：

1. 单位奖励界面，能否像商队事件的“奖励单位”选项或某些单位相关选项一样，在选项下方显示奖励单位兵种；
2. 需要考察如果玩家没有购买对应dlc，如何避免奖励对应dlc提供的单位；

本阶段可按下面优先级执行：

1. 必须完成敌军生成派系接管。
2. 必须完成装备奖励派系接管。
3. 单位奖励第三选项若改动成本过高，可延后到后续阶段或阶段 D 的后续增量。

开发者 agent 不应为了第三项拖慢整个阶段 D 主闭环。

## 节点初始化规则

阶段 D 需要定义 run 的起始节点。

第一版建议：

- 玩家 run 初始节点固定为震旦。

原因：

- 当前玩家主派系就是震旦。
- 这能减少阶段 D 第一版的初始状态分叉。

## 候选池构建与离线脚本

由于阶段 D 会开始涉及大量不同派系的单位池与装备池，允许使用离线脚本辅助生成 Lua 配置。

建议原则：

1. 离线脚本负责“扫描 DB -> 生成 Lua 配置表”。
2. 运行时脚本负责“读取 Lua 配置 -> 参与随机逻辑”。
3. 不要把大规模 DB 解析逻辑塞进运行时 Lua。

若使用 Python：

- 按 `powershell-python-fix` 的约定执行。
- 默认使用 `D:\\SoftWares\\miniforge3\\python.exe`。
- 不依赖裸 `python` 命令。

建议第一版离线生成的内容类型：

- 多派系单位池 Lua
- 多派系装备池 Lua
- 节点派系列表 Lua

## 本地化要求

阶段 D 会新增一组目的地选择事件文本。

最低要求：

- 事件标题非空
- 事件正文非空
- 两个随机目的地选项能看懂是哪个派系
- `留在当前派系` 文本明确
- `稍后再选择` 文本明确

若条件允许，建议在正文里体现：

- 当前所在派系节点
- 这次可以前往的两个候选派系

## 日志要求

阶段 D 必须继续输出独立日志。

至少记录：

- 当前节点初始化结果
- 每次进入目的地选择时的当前节点
- 本次随机出的两个目的地节点
- 玩家是否选择留在当前派系
- 玩家是否选择稍后再选
- 最终节点切换结果
- 目的地恢复行为
- 敌军生成读取的是哪个节点派系
- 装备奖励生成读取的是哪个节点派系
- 单位池或装备池 fallback 的触发情况

## 开发者实现约束

开发者 agent 在阶段 D 中应遵守：

1. 先做节点与状态机，再接入多派系内容池，不要反过来。
2. 不要设计真实地图关系网，禁止引入复杂相邻图。
3. 多派系池配置应尽量数据驱动，不要在主逻辑里堆大量 `if faction == ...`。
4. 任何候选恢复语义都必须以“保存抽取结果”为原则，不能在恢复时重抽。
5. 若某派系内容池暂时缺失，应允许 fallback，而不是让主流程报错。
6. 若引入 Python 辅助脚本，执行方式必须符合 `powershell-python-fix` 的约定。

## 完成标准

阶段 D 完成后，应满足下面这些可观察结果：

1. 玩家完成装备奖励后，会进入目的地选择事件。
2. 目的地事件中能看到两个随机派系、留在当前派系、稍后再选择。
3. 玩家选择任一随机派系后，当前节点切换为该派系。
4. 玩家选择留在当前派系后，当前节点不变。
5. 玩家选择稍后再选择后，重新点击入口仍是同一组选项。
6. 存档并读档后，仍能恢复到同一组目的地选项或同一待选择状态。
7. 后续战斗敌军派系会跟随当前节点派系变化。
8. 后续装备奖励会读取“通用 + 玩家派系 + 当前节点派系”作为来源。
9. 若某节点派系池缺失，主流程不会中断，而是记录日志并 fallback。
10. 日志中能看出节点推进、候选生成、恢复与内容池接管流程。

## 对开发者 agent 的交付说明

开发者 agent 在正式编码前，应把实现理解锁定为：

- 阶段 D 不是“做地图系统大全”
- 阶段 D 是“做派系节点虚拟地图 v1”

如果实现过程中出现分歧，优先遵守以下排序：

1. 主流程稳定
2. 节点状态可恢复
3. 敌军与装备奖励成功接管节点派系
4. 再考虑扩充更多派系池与单位奖励联动

## 后续阶段衔接

阶段 D 完成后，阶段 E 再重点处理：

- 难度参数平衡
- 数据结构整理与模块化重构

也就是说，阶段 D 的职责是先让“地图推进”以最轻量方式成立，再把多派系内容逐步接上。
