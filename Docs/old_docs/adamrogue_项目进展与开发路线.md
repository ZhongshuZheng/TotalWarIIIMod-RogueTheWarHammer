## 已确认的玩法方向

根据 `Description.md`，本 Mod 的核心方向是“Rogue-like 商队闯关”，而不是常规扩张玩法。

当前已确认的基本共识：

1. 玩家持有一支特殊军队。
2. 玩家通过按钮或事件入口推进流程。
3. 事件循环以“单位奖励 -> 战斗 -> 装备奖励 -> 目的地推进”为核心。
4. 玩家正常养成领主/英雄，但不走常规大地图经营节奏。
5. 当前优先先做单派系、单人战役、可验证的玩法闭环。

## 产品目标拆分

正式版本至少要具备 6 个能力：

1. 玩家有一个稳定、可恢复、可继续推进的 run。
2. 玩家通过一个明确入口主动推进，而不是被动等回合开始自动弹窗。
3. 事件循环至少能完整覆盖：单位奖励 -> 战斗 -> 装备奖励 -> 目的地。
4. 难度会随进度推进，而不是固定模板。
5. 玩家失败后 run 会结束，并且能够重新开始。
6. 代码结构不再是单文件原型，而是能继续扩展的模块化实现。

## 推荐开发顺序

不建议现在同时铺开全地图、全派系、全奖励池。

推荐按下面顺序推进：

1. 阶段 A：正式入口与状态机收束
2. 阶段 B：正式战斗循环
3. 阶段 C：装备奖励事件
4. 阶段 D：虚拟地图与目的地推进
5. 阶段 E：游戏参数平衡与模块化重构
6. 阶段 F：针对游玩反馈问题进行逐一调整

## 当前设计决定

- 当前主支持派系仍是震旦。
- 当前开发目标仍以单人战役为主。
- 当前优先级是“玩法闭环”高于“高拟真复刻原版商队系统”。
- 当前日志系统保留独立文件输出，便于多 agent 协作排查。
- 阶段 B 的敌军生成应优先参考遭遇战模式或多人对战模式的单位价格，作为“兵种价值分”的来源。
- 阶段 C 的装备奖励不强制做内容过滤，任务专属、唯一物品与重复物品暂不排除。
- 阶段 C 的奖励应先进入派系库存，而不是直接自动装备到角色身上。
- 阶段 D 的地图节点以“主要派系”为单位，而不是设计复杂地理相邻关系图。
- 阶段 D 的目的地事件固定提供 4 类选择：两个随机派系节点、留在当前派系、稍后再选。
- 阶段 D 第一版不做真实地理转移，只记录当前节点派系，并将其作为后续内容生成上下文。
- 为了构建多派系单位池与装备池，允许引入离线脚本生成 Lua 配置；若使用 Python，默认按 `powershell-python-fix` 的约定直接调用 `D:\\SoftWares\\miniforge3\\python.exe`。


## 后续协作默认原则

- 优先复用原版商队相关脚本模式。
- 所有 Lua API 先在 `OriginalGameData` 中检索确认，再正式写入 Mod。
- DB / 本地化 / Lua 三部分同步设计，避免只写脚本不补表。
- 每次只推进一个可验证功能块，避免一次性铺太大。
- 阶段级实现必须先有规范文档，再交给开发者 agent。
